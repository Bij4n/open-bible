Without the snapshot, `after_destroy_commit` would find no groups and no broadcast would fire.

- **2026-04-19 Sprint 6** — **`User#author_name` = display_name → email local-part.** Centralises public attribution; full email never leaves the server for display. Retrofitted onto Sprint 4's group members list and note attribution for consistency.
- **2026-04-19 Sprint 6** — **Partial `viewer` local with `current_user rescue nil` fallback.** Broadcast renders have no Warden in the env, so `current_user` raises `Devise::MissingWarden`. The rescue lets the partial render without author controls during broadcasts (correct — non-authors shouldn't see edit/delete anyway) and falls through to `current_user` in normal view contexts.

- **2026-04-19 Sprint 7** — **Separate `/public/bible/...` route for the community view.** `/bible/...` remains the signed-in personal reader; signed-out visitors at `/bible/...` redirect to the public variant. Preserves Sprint 3-6 semantics, matches the existing `/groups/:id/bible/...` namespacing, and avoids rewriting the personal-reader specs.
- **2026-04-19 Sprint 7** — **Boolean `users.admin` column; admin seeded via `bin/rails runner`.** Smallest footprint for a single-role system; enum migration is trivial when/if moderator + editor roles appear. `ensure_admin` heads `:not_found` (not 403) so admin route existence isn't advertised.
- **2026-04-19 Sprint 7** — **Both flagging and soft-delete, independent workflows.** Users flag via `Flag` (polymorphic Note/Comment, reason enum, idempotent per-user); admins moderate via `notes.hidden_at` / `hidden_by_id`. Admin can hide without a flag; admin can resolve a flag without hiding. The two channels are orthogonal.
- **2026-04-19 Sprint 7** — **Upvotes only, no downvotes.** Religious content attracts brigading; positive-signal-only is the community norm we want. `sorted_for_public` is featured desc → upvote count desc → created_at desc, with the count pulled in via a correlated subquery so featured/non-featured ties break by popularity without a full outer join.
- **2026-04-19 Sprint 7** — **Per-note `featured` boolean; pin-to-top of the public bible chapter view.** No separate "featured notes" list surface; that's a different feature (editorial roundup) for another sprint.
- **2026-04-19 Sprint 7** — **`Note.visible_to(user)` short-circuits to `all` for admins.** Admins see every note (hidden, private, group-shared) from one unified scope — the admin interface piggybacks on the same query. Non-admins get the four-branch union that excludes hidden-public.
- **2026-04-19 Sprint 7** — **No DB-level FK on `hidden_by_id` / `featured_by_id`.** Moderation audit should outlive the admin's account deletion; referential purity loses to the audit trail here. `belongs_to ... optional: true` captures the model-level association without the cascade risk. strong_migrations' concurrent-index requirement also pushed us to split the FK from the reference.
- **2026-04-19 Sprint 7** — **Admin::UsersController deferred.** Admin-promote/demote happens via `bin/rails runner User.find_by(email: ...).update!(admin: true)`. Single-admin reality at current scale; the UI surface isn't worth its specs yet.

- **2026-04-17 Sprint 8** — **Verse + note results in separated sections, verses first.** One visual hierarchy per type, rather than an interleaved relevance-merged list. Verses are stable public-domain content (canonical citations), notes are community commentary — merging them masked the distinction and made "showing top N" per-type scoping harder to explain.
- **2026-04-17 Sprint 8** — **Search is fully public (signed-out accessible).** Matches the public-domain nature of the content and the existing `/public/bible/...` surface. Note results go through `Note.visible_to(current_user)` (or `Note.public_visible` when anonymous), so no private content leaks regardless of auth state.
- **2026-04-17 Sprint 8** — **`Note.visible_to(current_user)` reused verbatim for search visibility.** The Sprint 4 scope already handles the four-branch union (own/direct/group/public) and the Sprint 7 admin short-circuit. Duplicating that logic into a search-specific scope was the obvious anti-pattern; the two-step pattern (pluck pg_search-matching ids, then filter through `visible_to`) lets us compose without tangling pg_search's `associated_against` JOINs with the union.
- **2026-04-17 Sprint 8** — **Top 20 per type, no pagination, "refine your search" copy instead.** At v1 scale a chapter-level result set is small and users refine faster than they page. A real pagination surface belongs after we see actual usage distributions.
- **2026-04-17 Sprint 8** — **Verse highlighting via ts_headline (`<mark>` wrapping), note highlighting via a plain-text custom helper.** Verse `body_text` is already plain, so pg_search's `with_pg_search_highlight` + `StartSel`/`StopSel` is safe and idiomatic. Action Text note bodies are HTML — letting ts_headline slice through a rich document would mangle tags — so `SearchHelper#highlight_terms` takes `body.to_plain_text`, escapes it, and wraps each term manually.
- **2026-04-17 Sprint 8** — **Action Text search via `associated_against: { rich_text_body: [:body] }`, not `against: :body`.** Action Text's `has_rich_text :body` creates a `rich_text_body` association pointing at the `action_text_rich_texts` table. `against: :body` on the Note model would look for a column that doesn't exist. The `associated_against` form is the pg_search idiom for joining through a belongs_to and searching a column on the joined table.
- **2026-04-17 Sprint 8** — **Known caveat on note search: HTML tag names match as tokens.** Since Action Text stores rendered HTML (`<p>`, `<strong>`, `<em>`), a search for "strong" hits the tag as well as the word. Acceptable for v1 — the ranking still surfaces actual matches first, and the plain-text preview hides the tags in the UI. If this becomes a real problem, a before_save hook that mirrors the plain-text into a dedicated searchable column is the next step.

- **2026-04-19 Sprint 9** — **JSON-text storage + in-Ruby cosine, not pgvector.** The pgvector system package requires host-level `sudo apt install`; we pivoted to `text` column storing JSON arrays and compute cosine similarity in Ruby over the ~31k-row KJV corpus. Swapping to pgvector later is a column-type migration plus a rewrite of `VerseEmbedding.search_by_similarity`; nothing upstream (Rails client, `SemanticSearchService`, controller, rake task, view) needs to change. The in-Ruby path is ~1–3s per query on the full corpus — acceptable for v1 but flagged as the first optimization target if semantic search proves valuable.
- **2026-04-19 Sprint 9** — **`all-MiniLM-L6-v2` (sentence-transformers, 384-dim) for embeddings.** Quality is good enough for devotional / topical queries, 80MB download fits a laptop comfortably, CPU inference is ~14k sentences/sec. Dimensions chosen with the 31k-row corpus in mind — mpnet-768 would double storage and halve throughput with marginal quality gain. Swapping models is a rake-task re-run.
- **2026-04-19 Sprint 9** — **Python service runs host-native under `bin/embedding`, wired into `Procfile.dev`.** No Docker Compose — the existing dev workflow is native (Puma + Tailwind watcher + Solid Queue), and introducing Compose for one service would invert that. `bin/embedding` bootstraps `.venv` on first run, pip-installs requirements, and exec's uvicorn. `EMBEDDING_SERVICE_SKIP=1` no-ops the process so non-search work doesn't require the 500MB torch download. Production deployment is explicitly deferred; if semantic search proves valuable in use, that gets its own sprint.
- **2026-04-19 Sprint 9** — **Verses-only embedding scope for v1.** Note embeddings would require live invalidation (re-embed on every Note update, plus a post-retrieval `Note.visible_to` filter since pgvector — or our Ruby equivalent — knows nothing about the four-branch visibility union). Deferred; semantic verse search alone is already the main user value. The UI warns when scope=notes + mode=semantic (`search.semantic_notes_pending`) rather than silently returning empty results.
- **2026-04-19 Sprint 9** — **Mode is a toggle on the same query, not a tab or smart fallback.** `?mode=keyword|semantic` slots cleanly into Sprint 8's `?scope=` parameter pattern. Unknown values default to keyword so a malformed bookmark can't break the page. "Combined/merged results" was rejected because pg_search's ts_rank and cosine similarity aren't comparable without hand-tuned calibration we can't do yet. "Smart fallback" was rejected because it hides the feature.
- **2026-04-19 Sprint 9** — **Graceful degradation on service outage.** If `EmbeddingService.healthy?` returns false on a semantic query, the controller returns keyword results merged with `semantic_fallback: true`, and the view shows a non-fatal banner. `SemanticSearchService` also rescues `EmbeddingError` into `{ available: false }` so a transient post-healthcheck failure behaves the same way. No 500s, no dead-end UX.
- **2026-04-19 Sprint 9** — **`Net::HTTP` over HTTParty for the Rails client.** One fewer gem, Rails stdlib is sufficient for two JSON endpoints. Timeouts are per-method (5s health, 30s embed) since embed latency scales with batch size. All transport failures (parse, socket, timeout) collapse into a single `EmbeddingError` so callers don't have to know the transport.
- **2026-04-19 Sprint 9** — **WebMock in the test group + `allow_localhost`.** Service-client specs stub the HTTP boundary rather than depending on the Python service running. `disable_net_connect!(allow_localhost: true)` means any forgotten real-world request shows up as a loud test failure while Capybara / Selenium drivers keep working.
- **2026-04-19 Sprint 9** — **Similarity surfaced as "N% match" badge, not raw distance.** Users think in percentages; cosine similarity happens to range 0–1 for normalized embeddings so `(score * 100).round` reads right. Semantic results attach a `similarity_score` singleton to each returned `Verse`; the keyword path has no such attribute, so the view branches on `respond_to?(:similarity_score)`.

- **2026-04-20 Sprint 10** — **RV1909 source is gratis-bible/bible/es/sparv.xml, identified by text-level provenance.** seven1m/open-bibles only ships Spanish in USFX (Haiola isn't run on their Spanish sources), so the "same provenance chain as KJV" plan from the confidence report didn't hold. gratis-bible publishes Spanish OSIS via ZefToOsis, but the file is labeled only "Spanish Reina-Valera" with no year. Three text markers — Gen 1:1 `crió`, Ps 23:1 unaccented `JEHOVA`, John 3:16 archaic `á su Hijo unigénito` — match eBible.org's published RV1909 sample exactly and rule out RVA (1989), SEV (1865), and the copyrighted RVR1960. SHA pinned; future-us has enough docstring to re-verify if the file ever changes.
- **2026-04-20 Sprint 10** — **OsisImporter generalised to handle both milestone and container OSIS dialects.** seven1m/KJV uses milestone-style (`<verse sID/>text<verse eID/>`); gratis-bible/RV1909 uses container-style (`<verse osisID=…>text</verse>`). The SAX handler now branches on attribute shape (`sID` → milestone, `osisID` alone → container) and `end_element` closes container verses/chapters. Synthetic `container_mini.xml` fixture covers the new path; existing `kjv_mini` tests were unaffected. The decision to split this into its own commit (before the RV1909 integration commit) paid off — each commit has a single responsibility and the dialect-agnostic importer is a general capability worth testing independently.
- **2026-04-20 Sprint 10** — **URL slugs are OSIS codes, not localized names.** `/bible/rv1909/john/3` for a Spanish user too; the heading renders "Juan 3" but the URL stays translation-neutral. The brief proposed `/bible/rv1909/juan/3` with a fallback lookup against `name_es`, which would have required locale-aware routing, disambiguation logic for overlapping names ("San Juan" vs "John"), and two collation-dependent queries. Trade is a minor aesthetic cost (Spanish readers see an English-ish URL) for a significant correctness and simplicity win (single resolution path, no LIKE against localized names).
- **2026-04-20 Sprint 10** — **Book display name is `I18n.locale`-driven, not translation-driven.** Because the RV1909 reader is valid for an English-locale user too, we pick `name_en` or `name_es` off the Book based on the user's UI locale, not on the translation's language. A Spanish user reading the KJV sees "Juan 3"; an English user reading the RV1909 sees "John 3". Keeps the display decision aligned with one axis (UI locale) rather than tangling language-of-interface with language-of-content.
- **2026-04-20 Sprint 10** — **Red-letter mirror ships only fully-red verses.** Character offsets can't cross translations — "For God so loved the world" (108 chars) doesn't align with "Porque de tal manera amó Dios al mundo" (150 chars). `Bible::RedLetterMirror` transfers verses whose KJV ranges are a single `[0, body_text.length]` span (i.e. uninterrupted Jesus discourse) and rewrites the range to the target verse's own length. Partial-red verses (Jesus words embedded mid-narrator-verse) get empty ranges in RV1909 — we'd be guessing where Spanish clauses split. Covers the bulk of practical value (John 14-17, Sermon on the Mount) cleanly; boundary verses lose red in RV1909, which is the honest trade.
- **2026-04-20 Sprint 10** — **Translation selector is a picker reusing the chapter-picker Stimulus pattern.** Option values are precomputed `bible_chapter_path` strings keyed on the current `osis_code`, so swapping translations Turbo-visits the equivalent URL in the target. Same behaviour as the existing chapter-picker: `event.target.value -> Turbo.visit`. Five-line controller per concern rather than shoving both roles under one generic name — readability per file matters more than DRY at this scale.
- **2026-04-20 Sprint 10** — **`/bible` route goes through `Bible::Reader#entry`, not a static redirect.** Sprint 2 added `users.default_translation_id` but nothing ever used it. ### Sprint 15 — Homepage content + Bitcoin donations + SMTP wiring

**Goal:** close out the v1 product surface. Rewrite the homepage content (hero, features, how-it-works, about, donation CTA) on top of the Sprint 13 visual shell. Ship a static-wallet Bitcoin donation flow — no processor, no KYC, no webhooks, no third-party service. Wire an SMTP provider so Devise password reset stops silent-failing. No visual-system work; the design pivot (Sprints 12-14) closed that.

**Decisions locked in this brief** (user confirms or vetoes before implementation commit):

- **Bitcoin donations: static wallet with manual rotation.** No OpenNode, no BTCPay, no processor of any kind. `/donate` renders the currently-active on-chain address with a QR (generated via the `rqrcode` Ruby gem — no external API, Rule 8 compliant) and a copy-to-clipboard Stimulus action. Admin rotates addresses via an admin UI behind the existing `ensure_admin` guard; rotation archives the previous address (never deleted — addresses always remain spendable on-chain, and the archive is an audit trail). User self-reports donations via a "I've sent a donation" button that POSTs to a confirmation endpoint; no on-chain verification in v1. Rationale for the pivot from OpenNode: user runs hardware + software wallets already, so address generation is not a new capability; a processor adds KYC, API-key management, webhook-secret management, and a third-party service that can deprecate or close the account — all misaligned with the self-hosted / no-Google / sovereignty posture established by Sprints 12-14. Scope fell from ~1 sprint to ~1 day. Upgrade path to xpub-derived rotation or a processor stays clean: the `BitcoinAddress.active` row is the interface the views read.
- **SMTP provider: Resend.** Rationale: 100/day + 3000/month free tier (confirmed 2026-04-24; 1 custom domain; 30-day retention). Modern DX, clean Rails setup via ActionMailer SMTP config. DNS verification required for `bible-together.org` sender (SPF + DKIM). Fallback pick if deliverability disappoints: Postmark.
- **Jurisdiction / entity status for donations** — **flagged as a blocker before the donation implementation commit**, not this brief. User resolves separately; affects the disclaimer copy on `/donate`. Brief drafts generic-disclaimer shape; copy revises when the answer lands.

**In scope:**

- **Homepage content rewrite** (`app/views/home/show.html.erb`): hero with single clear value proposition, features grid (character-level highlighting, private + shared notes, group Bible real-time collaboration, public notes with upvotes, bilingual KJV + RV1909, keyword search, dark mode, bilingual UI) — **only listing features that actually exist and work**, how-it-works section in ~3 steps, about section with mission/scope, donation CTA pointing at `/donate`. Bilingual copy in `config/locales/{en,es}.yml`. **No "open source" until the repo is public; no bare "accessibility-first" — "WCAG AA tested with axe-core" with the spec reference if we want the credential.** Semantic search is claimed only if the embedding pserv is verified live; given its status has been uncertain since 2026-04-21 and hasn't been re-checked, default to omitting it from the grid and adding it back in a follow-up commit once the pserv is confirmed green.
- **Bitcoin donations (static wallet)**:
  - `BitcoinAddress` model: `address` (string, not null), `active` (bool, default false), `archived_at` (datetime, nullable), `notes` (text — admin-only, e.g. "hardware wallet X / account 3 / addr 17"), `created_at` / `updated_at`. Partial unique index on `active` where `active = true` enforces at most one active row at a time at the DB level.
  - `DonationReport` model: `email` (nullable), `message` (nullable), `created_at`. One table, nullable PII — always log a row on `/donate/confirm` POST regardless of email presence, so `DonationReport.count` gives a usage signal and `DonationReport.where.not(email: nil).count` gives the email-bearing subset. Rejected the separate-pings-table alternative as avoidable fragmentation.
  - `DonationsController`:
    - `GET /donate` — renders the currently-active address + QR + copy-to-clipboard action + the generic disclaimer + the "I've sent a donation" confirm button. When no active address exists (never seeded / admin paused rotation mid-flight), renders a minimal "Donations temporarily unavailable" explainer page with no form.
    - `POST /donate/confirm` — accepts an optional email + optional short message via the form. Honeypot field (hidden input named `website`, CSS-hidden with `sr-only`, `tabindex="-1"`, `autocomplete="off"`) — if filled on submit, silently redirects to the thank-you page without persisting and without email capture. If honeypot empty, creates a `DonationReport` (email and message nullable, whichever the user supplied) and redirects to `/donate/thank-you`.
    - `GET /donate/thank-you` — renders acknowledgement copy; no state-dependent content.
    - When no active address exists, `POST /donate/confirm` returns 404 (consistent with "feature unavailable" — prevents curl POSTs from accumulating reports against a paused state).
  - `Admin::BitcoinAddressesController` behind `ensure_admin`: index lists all addresses (active + archived, newest first, with `notes`), `create` accepts a new address string + optional notes, `rotate` action archives the current active row and marks the new one active atomically in a transaction.
  - **Site-wide footer link to `/donate`** — low-key text link in the footer. When no `BitcoinAddress.active` exists, the footer link is hidden (not just disabled) to keep the "feature unavailable" state consistent across surfaces.
  - QR code rendering via the `rqrcode` gem (Ruby, MIT license, no external API, Rule 8 compliant). Rendered as inline SVG for zero additional HTTP requests.
- **SMTP wiring** — `config/environments/production.rb` `config.action_mailer.smtp_settings` pointing at Resend's SMTP endpoint. Credentials in Rails credentials (`smtp.resend.api_key`). Devise password reset confirmed working end-to-end via a request spec exercising the full mailer path. `default_url_options` host already `bible-together.org` (Sprint 12 deploy prep).

**Out of scope (defer to v2 / Sprint 16 or later):**

- Lightning address support (v1 is on-chain only — one payment rail, keep simple).
- xpub-derived per-visitor addresses (clean upgrade path preserved; marginal privacy gain not worth the crypto-lib surface area in v1).
- On-chain verification of self-reported donations.
- Rate limiting on `/donate/confirm` beyond the honeypot (monitor after launch; add cookie + IP throttle if spam becomes real).
- Donation amounts on `DonationReport` (user doesn't tell us how much; we don't guess).
- Donor dashboard, donor wall, receipt PDFs.
- Additional mailer templates beyond password reset (confirmation, unlock, email-changed — Devise's default templates render in the new auth_shell already).
- Custom error pages.
- Any changes to the 571-spec suite structure; this sprint adds specs, doesn't restructure.

**TDD approach per feature** (explicit, per Rule 9):

- **Donations**: request spec for `/donate` (active-address state renders the address + QR SVG + confirm form; no-active-address state renders the unavailable page without the form), request spec for `/donate/confirm` (honeypot-filled → redirect to thank-you with zero `DonationReport` rows written; honeypot-empty + email-present → creates `DonationReport` with email; honeypot-empty + email-absent → creates `DonationReport` with email nil; no-active-address state → 404), request spec for `Admin::BitcoinAddressesController` (non-admin → 404, admin rotate → active flag swaps atomically, archived row keeps original created_at + gains `archived_at`). System spec for `/donate`: QR SVG is present in the DOM, copy-to-clipboard Stimulus action fires on click, footer donate link appears on any page when active address exists and doesn't appear when it doesn't. i18n coverage spec: both locales render all donation-related strings.
- **Mailer**: ActionMailer preview specs for both locales render without errors; delivery spec asserts `deliver_now` enqueues one mail; Devise reset-password request spec exercises the full flow and asserts the delivered email subject + body + reset-link host.
- **Homepage**: system specs for every CTA — "Read the Bible" lands on `/public/bible/kjv/gen/1`, footer "Donate" lands on `/donate`, every feature-card link lands where it claims. Request spec for i18n coverage in both locales so a missing translation can't ship silently.

**Operational baseline:** first sprint where Rule 9 applies from commit 1. Every UI-touching commit ships its matching system spec in the same commit; branch protection on main refuses non-green pushes. `spec/rails_helper.rb` rebuilds Tailwind before any system-spec run (commit 276cc85), so stale-CSS false-green can't re-occur. QR generation uses `rqrcode` (Ruby, self-hosted) — no external QR-as-a-service API (Rule 8). Admin rotation UI inherits `ensure_admin` from `ApplicationController` (Sprint 7).
